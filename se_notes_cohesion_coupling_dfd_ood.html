Cohesion

Definition: Cohesion measures how strongly the elements inside a single module are related to each other. A highly cohesive module does one thing and does it completely.

High cohesion → easier to understand, maintain, and reuse. Low cohesion → module is a "grab-bag" of unrelated functions.

Types of Cohesion (worst → best)

#TypeDescriptionRating
1CoincidentalElements grouped randomly with no logical relationVery Low
2LogicalElements perform similar operations selected by control flag (e.g., all I/O)Low
3TemporalElements execute at the same time (e.g., system startup)Low-Med
4ProceduralElements follow a certain sequence of executionMedium
5CommunicationalElements operate on the same data setMedium-High
6SequentialOutput of one element is input to nextHigh
7FunctionalEvery element contributes to a single, well-defined taskVery High ✓

Detailed Examples

1. Coincidental Cohesion (worst)

// Completely unrelated functions dumped together
module Utils {
  printReport()
  calculateTax()
  sendEmail()
  parseDate()
}

No logical connection between functions.

2. Logical Cohesion

// All do "input", chosen by type flag
module InputHandler {
  handleInput(type) {
    if (type == "keyboard") ...
    if (type == "mouse") ...
    if (type == "file") ...
  }
}

Related by category, but uses control flags.

3. Temporal Cohesion

// All happen at startup — not logically related
module Startup {
  initDB()
  loadConfig()
  startLogger()
  openSocket()
}

Related only by when they run.

4. Procedural Cohesion

// Sequence matters, but data not shared
module FileOps {
  openFile()
  readPermissions()
  closeFile()
  logAccess()
}

Ordered steps, but different data items.

5. Communicational Cohesion

// All work on same customer record
module CustomerOps {
  getCustomerName(id)
  getCustomerEmail(id)
  getCustomerAddress(id)
}

Same data but different operations.

6. Sequential Cohesion

// Output of step feeds next step
module DataPipeline {
  rawData = readCSV()
  cleaned = cleanData(rawData)
  result = analyzeData(cleaned)
  return result
}

Each element's output is next one's input.

7. Functional Cohesion (best)

// Every line contributes to ONE task: compute invoice total
function calculateInvoiceTotal(items, taxRate) {
  subtotal = sum(items.price)
  tax = subtotal * taxRate
  return subtotal + tax
}

Single, well-defined purpose. Every element is essential and related.

Aim for Functional or at least Sequential cohesion. Avoid Coincidental and Logical cohesion at all costs.

Coupling

Definition: Coupling measures the degree of interdependence between modules. Low coupling means modules are largely independent and can be developed, tested, and modified separately.

Low coupling → easier maintenance, better reuse, isolated failures. High coupling → ripple effect — changing one module breaks many others.

Types of Coupling (worst → best)

#TypeHow Modules InteractRating
1ContentModule A directly modifies internals of Module BWorst
2CommonModules share global dataVery High
3ExternalModules share externally imposed format/protocolHigh
4ControlOne module passes a control flag to dictate behavior of anotherMedium-High
5Stamp (Data Structure)Modules share composite data structure but use only part of itMedium
6DataModules share only simple data via parametersLow ✓
7Message / No CouplingModules communicate via messages; completely independentBest

Detailed Examples

1. Content Coupling (worst)

// Module A directly accesses Module B's internals
moduleA {
  moduleB.privateVar = 42  // ← direct access!
  goto moduleB.label_5     // ← goto into another module!
}

2. Common Coupling

// Both modules use a global variable
int globalCounter = 0;

moduleA() { globalCounter++; }
moduleB() { globalCounter--; }
// If counter goes wrong, who caused it?

3. Control Coupling

// Flag controls behavior of called module
moduleA() {
  process(flag = "PRINT")   // ← control flag!
}
process(flag) {
  if(flag == "PRINT") printData()
  if(flag == "SAVE")  saveData()
}

4. Stamp Coupling

// Passing whole Employee object, using only name
moduleA() {
  emp = getEmployee()
  printName(emp)  // ← passes whole record
}
printName(Employee emp) {
  print(emp.name) // uses only .name
}

5. Data Coupling (desirable)

// Only needed data is passed as simple params
calculateArea(length, width) {
  return length * width
}
// Called as: calculateArea(5, 10)
// No hidden dependencies, no global state

6. Message Coupling (best)

// Modules communicate via messages/events
EventBus.publish("order.placed", {orderId: 42})

// OrderModule subscribes independently
EventBus.subscribe("order.placed", sendEmail)
// Total independence — neither knows the other

Cohesion vs Coupling Summary

PropertyDirectionWithin/BetweenGoal
Cohesion↑ Higher = BetterWithin a moduleMaximize
Coupling↓ Lower = BetterBetween modulesMinimize
The golden rule: High Cohesion + Low Coupling = well-designed modular system.

Data Flow Diagram (DFD)

Definition: A graphical representation of how data flows through an information system. DFDs show inputs, outputs, data storage, and the processes that transform data — without showing control flow or decision logic.

DFD Notations

Rectangle — External entity (person/system outside boundary)
Circle/Ellipse — Process (transforms data, has verb label)
Open Rectangle — Data Store (file, DB, register)
Labeled Arrow — Data Flow (direction of data movement)

DFD Levels

Level 0

Context Diagram

Single process bubble representing entire system. Shows only external entities and data flows in/out. No internal detail.

Level 1

Level 1 DFD

Breaks Level 0's single process into 4-7 major sub-processes. Data stores introduced. Major data flows shown.

Level 2

Level 2 DFD

Further decomposes each Level 1 process into detailed sub-processes. Most detailed representation.

Example: Online Order System

Level 0 — Context Diagram

Level 1 — Expanded

DFD Rules

  • Processes must have at least one input and one output
  • Data stores must be connected to a process (never directly to external entities)
  • External entities cannot connect directly to data stores
  • Data flows must be labeled with the data being transferred
  • No control flow — only data flow is shown

Refactoring of Design

Definition: Refactoring is the process of restructuring existing code/design without changing its external behavior. The goal is to improve internal structure — readability, maintainability, extensibility — while keeping functionality identical.

"First make it work, then make it right."

Why Refactor?

Benefits

  • Reduces technical debt
  • Improves code readability
  • Easier to add new features
  • Reduces bug density
  • Improves performance
  • Easier unit testing

Code Smells (signals to refactor)

  • Long methods / god classes
  • Duplicate code
  • Magic numbers
  • Deep nesting
  • Poor variable names
  • Tight coupling

Common Refactoring Techniques

1. Extract Method

Before:

void printReport() {
  // print header
  System.out.println("=====");
  System.out.println("Report");
  System.out.println("=====");
  // print body
  for(item in items) print(item);
}

After:

void printReport() {
  printHeader();
  printBody();
}
void printHeader() {
  System.out.println("=====");
  System.out.println("Report");
  System.out.println("=====");
}

2. Replace Magic Numbers with Constants

// Before
if (temp > 100) boil();
// After
final int BOILING_POINT = 100;
if (temp > BOILING_POINT) boil();

3. Extract Class (Split God Class)

// Before: one huge class
class Order {
  calculateTotal()
  sendEmail()
  printInvoice()
  validatePayment()
  updateInventory()
}
// After: each class has one job
class Order { calculateTotal() }
class Notifier { sendEmail() }
class Invoice { print() }
class PaymentGateway { validate() }
class InventoryManager { update() }

4. Replace Conditional with Polymorphism

// Before
double getSpeed(Bird b) {
  if (b.type == "EUROPEAN") return 35;
  if (b.type == "AFRICAN") return 40;
  if (b.type == "PARROT") return 60;
}
// After
abstract class Bird { abstract speed(); }
class EuropeanBird { speed() { return 35; }}
class AfricanBird   { speed() { return 40; }}
class Parrot        { speed() { return 60; }}

Refactoring in Design (not just code)

Design-Level Smells

  • Divergent change (many reasons to change one class)
  • Shotgun surgery (one change affects many classes)
  • Feature envy (method uses another class's data too much)
  • Inappropriate intimacy (classes know too much about each other)

Design Refactoring Actions

  • Move method/field to more appropriate class
  • Extract interface from class
  • Replace inheritance with delegation
  • Introduce design pattern (Strategy, Observer, Factory)
Always run tests before and after refactoring. Refactoring without tests is risky — you may break behavior unknowingly.

Object-Oriented Design (OOD)

Definition: OOD is a design approach that models a system as a collection of interacting objects. Each object encapsulates data (attributes) and behavior (methods). Design uses principles like inheritance, polymorphism, and encapsulation.

Core OOP Concepts

Encapsulation

Bundling data + methods. Hides internal state. Access via public interface only.

class BankAccount {
  private balance  // hidden
  deposit(amt) { balance += amt }
  getBalance() { return balance }
}

Inheritance

A class acquires properties/methods of another. Promotes code reuse.

class Animal { speak() }
class Dog extends Animal {
  speak() { return "Woof" }
}

Polymorphism

Same interface, different behavior. A Dog and Cat both speak() differently.

Animal[] animals = [Dog, Cat, Bird]
for (a in animals) a.speak() // different output

Abstraction

Exposing only essential features. Hiding complexity.

abstract class Shape {
  abstract area()  // contract
}
class Circle extends Shape {
  area() { return π * r² }
}

SOLID Principles

LetterPrincipleMeaning
SSingle ResponsibilityA class should have only one reason to change
OOpen/ClosedOpen for extension, closed for modification
LLiskov SubstitutionSubtypes must be substitutable for their base types
IInterface SegregationMany specific interfaces better than one general interface
DDependency InversionDepend on abstractions, not concretions

OOD Process Steps

  1. Identify objects from requirements (nouns → classes)
  2. Define attributes for each object
  3. Define operations/methods (verbs → methods)
  4. Identify relationships (association, aggregation, composition, inheritance)
  5. Design interfaces between objects
  6. Apply design patterns where applicable
  7. Refine with UML diagrams (Class, Sequence, etc.)

OOD vs Function-Oriented Design

AspectOODFunction-Oriented
Basic unitObjectsFunctions / procedures
DataEncapsulated in objectsPassed as parameters
StateDistributed in objectsShared / global
DecompositionBy objects/classesBy functions/modules
ReuseInheritance, compositionLibraries, includes
Best forComplex, evolving systemsData transformation, scientific

Function-Oriented Design

Definition: Also called Structured Design or Procedural Design. The system is decomposed into a set of functions/modules that operate on data. Each module performs a specific computation. Based on top-down decomposition and stepwise refinement.

Key Concepts

Top-Down Decomposition

Start with the overall problem. Repeatedly break it into smaller sub-problems until each is small enough to implement directly.

Main Program
├── InputModule
│   ├── ReadData()
│   └── ValidateInput()
├── ProcessModule
│   ├── Calculate()
│   └── ApplyRules()
└── OutputModule
    ├── FormatResult()
    └── PrintReport()

Stepwise Refinement

Start with abstract description, progressively add detail.

// Level 1 (abstract)
ProcessOrder()

// Level 2 (refined)
ProcessOrder() {
  ValidateOrder()
  CheckInventory()
  ProcessPayment()
  ShipOrder()
}

// Level 3 (detailed)
ValidateOrder() {
  CheckFields()
  VerifyCustomer()
}

Use Case Diagram

Definition: A Use Case Diagram shows the functional requirements of a system from the end-user's perspective. It captures who (actors) uses the system and what (use cases) they can do with it.

Notations

Relationships

RelationshipNotationMeaning
AssociationSolid lineActor participates in use case
IncludeDashed arrow «include»Use case always calls another (mandatory sub-task)
ExtendDashed arrow «extend»Use case conditionally extends another (optional behavior)
GeneralizationSolid arrow with hollow headActor/use case inherits from another

Example: Online Banking System

«include» — Login always includes Authenticate 2FA (mandatory)
«extend» — Transfer Funds may optionally extend to Send OTP (conditional on high amount)
Association — Customer can Login, Transfer, View Statement; Admin can Manage Accounts

Sequence Diagram

Definition: A Sequence Diagram shows how objects interact in a time sequence. It emphasizes the order of messages exchanged between lifelines (objects/actors) to accomplish a use case or scenario.

Notations

ElementNotationMeaning
LifelineBox at top + dashed vertical lineAn object or actor participating in interaction
Activation BarThin rectangle on lifelinePeriod during which object is active/executing
Synchronous MessageSolid arrow →Caller waits for response (method call)
Return MessageDashed arrow ← - -Return value from a method
Asynchronous MessageOpen-headed arrow →Caller doesn't wait (fire & forget)
Self CallArrow looping back to same lifelineObject calling its own method
opt frameBox labeled «opt» with conditionOptional interaction
loop frameBox labeled «loop»Repeated interaction
alt frameBox labeled «alt» with dashed dividerAlternative paths (if-else)

Example: User Login Sequence

Key Points to Remember

Time flows downward

Messages are ordered top to bottom. Earlier interactions are higher on the diagram.

Combined Fragments

  • alt — if/else alternatives
  • opt — optional (if, no else)
  • loop — repeated interaction
  • par — parallel execution
  • ref — reference to another diagram
Use Sequence Diagrams to design and verify the dynamic behavior of a use case. Each use case scenario typically gets one sequence diagram.
