Software Engineering detailed notes on Validation/Verification, Testing, Project Estimation, and Risk Management

Validation & Verification
Core Concept The V&V Distinction
Verification — "Are we building the product right?"

Checks that the software conforms to its specification. Internal process-oriented. Involves inspections, walkthroughs, reviews. No actual execution needed.

Validation — "Are we building the right product?"

Checks that the software meets the customer's actual needs. External user-oriented. Involves testing on real or simulated inputs. Software must be executable.

Barry Boehm coined the classic distinction. Both V&V activities run throughout the software lifecycle — not just at the end.

Verification Activities
Requirements Review
Check specs for completeness, consistency, and feasibility early on.
Design Walkthrough
Peer inspection of architecture and detailed design documents.
Code Inspection
Structured reading of source code against standards (e.g. Fagan inspection).
Static Analysis
Tool-based analysis (linters, SAST tools) without executing code.
Formal Verification
Mathematical proof that code satisfies its formal specification (e.g. model checking).
Validation Activities
Unit Testing
Execute individual modules in isolation to verify logic.
Integration Testing
Execute combined modules to verify interfaces and interactions.
System Testing
Execute the full system against all functional and non-functional requirements.
Acceptance Testing
User/client tests to confirm the system satisfies real business needs (UAT).
Regression Testing
Re-execute test suites after changes to catch unintended side effects.
Key Principle The V-Model

The V-Model maps each development phase to a corresponding testing phase. Each left-side phase produces a document/artifact that directly drives the right-side validation:

Development Phase (Left)↔Testing Phase (Right)
Requirements Analysis↔Acceptance Testing
System Design↔System Testing
Architecture Design↔Integration Testing
Module Design↔Unit Testing

Testing Types
White-Box Testing

Also called glass-box, clear-box, or structural testing. Tester has full knowledge of the internal code structure, logic, and implementation.

Tests exercise specific paths, branches, conditions, and loops within the source code.

Statement coverage
Every line of code is executed at least once.
Branch coverage
Every true/false branch of each decision is executed.
Path coverage
Every independent path through code is exercised (basis path testing).
Condition coverage
Each boolean sub-expression is evaluated to T and F.
Basis Path TestingCyclomatic ComplexityLoop TestingData Flow Testing
Black-Box Testing

Also called behavioral or functional testing. Tester has no knowledge of internal code. Focus is solely on input–output behavior against specifications.

Tests are derived from requirements and specifications, not from code structure.

Equivalence Partitioning
Divide input domain into classes; one test per class covers all values in it.
Boundary Value Analysis
Test at and just inside/outside the edges of input partitions.
Decision Table Testing
Tabulate combinations of conditions and corresponding actions.
State Transition
Test state changes triggered by inputs/events.
Error Guessing
Experience-based ad-hoc testing of likely error-prone areas.

Unit Testing

The smallest testable unit (function, method, class) is tested in isolation. Dependencies are replaced with stubs (simplified callees) or drivers (simplified callers) so each module is tested independently.

Scope
Individual module or function, tested in isolation from the rest of the system.
Who performs
Typically the developer who wrote the code (sometimes separate QA). Often automated (JUnit, PyTest, Jest, etc.).
Stubs
Dummy modules that replace callees — return hard-coded values so the unit under test can run.
Drivers
Test harnesses that simulate the caller, passing inputs and capturing outputs.
Goal
Verify logic, computation, boundary conditions, exception handling, and local data structures within the module.
Integration Testing

Tests how modules work together. Combines verified units and checks data flow and interface contracts across module boundaries.

Big Bang
All modules integrated at once — hard to isolate failures.
Top-Down
Start from top-level modules, stubs replace lower modules. Breadth-first or depth-first.
Bottom-Up
Start from lowest modules, drivers replace higher. Useful when low-level components are complex.
Sandwich
Combines top-down and bottom-up simultaneously to reduce time.
System Testing

Tests the complete, integrated system against the SRS. Covers both functional and non-functional requirements.

Functional
Verify all specified features and use cases behave correctly end-to-end.
Performance
Load, stress, and scalability testing under realistic and peak conditions.
Security
Penetration testing, authentication, authorization, and data integrity checks.
Usability
UI and UX evaluation — learnability, efficiency, error recovery.
Reliability
Mean time between failure (MTBF) and mean time to repair (MTTR) testing.
Regression Testing

After any modification (bug fix, new feature, refactoring), a selected subset of existing tests is re-executed to ensure no previously passing functionality has broken.

Why needed
Changes in one module can unexpectedly break other modules due to dependencies.
Test selection
Retest all (safe but slow), test selection (risk-based), test prioritization (critical tests first).
Automation
Regression suites must be automated (CI/CD pipelines) to be practical in agile environments.
Use Case Testing

Tests derived directly from use cases in the requirements. Each use case is executed scenario-by-scenario including alternate and exception flows.

Main flow
The happy-path scenario where all preconditions are met and the goal is achieved.
Alternate flows
Valid variations of the main flow (e.g., choose a different payment method).
Exception flows
Error conditions — invalid input, system unavailability, timeout handling.
Traceability
Each test maps back to a specific use case, ensuring requirement coverage is measurable.
Project Estimation Techniques
COCOMO — Basic Model

COnstructive COst MOdel (Boehm, 1981). Estimates effort and duration from estimated source lines of code (KLOC = thousands of lines of code). Three project modes:

Effort (Person-Months)
E = a × (KLOC)^b

Duration (Months)
D = c × E^d

People Required
P = E / D

Intermediate COCOMO

Refines the Basic COCOMO by applying 15 Cost Driver Attributes (EAF — Effort Adjustment Factor) that reflect real-world project differences not captured by KLOC alone.

Intermediate COCOMO Formula
E = a × (KLOC)^b × EAF

where EAF = product of all cost driver multipliers
EAF = CD₁ × CD₂ × ... × CD₁₅
CategoryCost Drivers (examples)
Product AttributesRequired reliability, database size, product complexity
Hardware AttributesExecution time constraint, main storage constraint, virtual machine volatility
Personnel AttributesAnalyst capability, programmer capability, virtual machine experience, language experience
Project AttributesUse of software tools, modern programming practices, required schedule

Each cost driver is rated on a scale (Very Low → Extra High). Rating maps to a multiplier (e.g., Very Low = 1.40, Nominal = 1.00, Very High = 0.70 for analyst capability). EAF ranges typically from 0.9 to 1.4 in practice.

Function Point Analysis (FPA)

Albrecht (1979). Measures software size in function points — based on functionality delivered to the user, independent of technology or programming language.

Step 1: Count Unadjusted Function Points (UFP)
UFP = Σ (Count × Weight) for each of the 5 component types

Step 2: Apply Technical Complexity Factor (TCF)
TCF = 0.65 + 0.01 × Σ(Fi)    where Fi = 14 General System Characteristics rated 0–5

Step 3: Adjusted Function Points
AFP = UFP × TCF

Delphi Technique

Expert judgment-based estimation. A facilitator collects independent estimates from multiple experts anonymously, shares the aggregated results, and iterates until consensus is reached.

Step 1
Coordinator presents problem and specifications to all experts independently.
Step 2
Each expert submits an anonymous estimate with rationale.
Step 3
Coordinator summarizes results and redistributes (anonymously) to all experts.
Step 4
Experts revise estimates considering others' reasoning. Repeat until convergence.
Advantage
Avoids groupthink; leverages diverse expertise; useful when historical data is scarce.
Wideband Delphi
Extended version used in software where group discussion is allowed between rounds.
PERT / CPM

Program Evaluation and Review Technique (PERT) and Critical Path Method (CPM) are network-based scheduling and estimation techniques.

PERT Three-Point Estimate
tE = (O + 4M + P) / 6
σ = (P − O) / 6
Variance = σ²

where
O = Optimistic  |  M = Most Likely  |  P = Pessimistic
Critical Path
Longest path through the activity network — determines minimum project duration.
Float / Slack
Time an activity can be delayed without delaying the project. Critical path has zero float.
CPM vs PERT
CPM uses single deterministic estimates; PERT uses probabilistic 3-point estimates for uncertainty.
Software Risks & Risk Management
What is Risk?

A risk is an uncertain event or condition that, if it occurs, has a negative effect on a project's objectives (scope, schedule, cost, quality). Risk = Probability × Impact.

Project
Risks affecting schedule, resources, budget
Product
Risks affecting quality or performance of deliverable
Business
Risks affecting the organization or market
Technical
Risks from technology choices or complexity
Reactive Risk Management
Also called "fire-fighting" or crisis management

Team waits until a risk manifests as a problem before taking action.

Approach
React to problems after they occur. No formal risk identification upfront.
Risk Mitigation
Also called "Indiana Jones approach" — manage crisis when it strikes.
Contingency
Buffers added at end. No systematic contingency plans per risk.
Drawback
Higher cost to fix, schedule overruns, panic-driven decisions, customer dissatisfaction.
When used
Small trivial projects, or teams with poor risk culture. Never recommended for critical systems.
Proactive Risk Management
Systematic, anticipatory approach

Risks are identified, analyzed, and mitigation strategies are planned before problems occur.

Step 1: Identify
Risk checklists, brainstorming, Delphi, decomposition, assumption analysis.
Step 2: Analyze
Qualitative (probability × impact matrix) and quantitative (Monte Carlo, EMV) analysis.
Step 3: Plan
Mitigation, avoidance, transference, or acceptance strategies per risk.
Step 4: Monitor
Track risk triggers and residual risks throughout the project lifecycle.
Step 5: Communicate
Risk register maintained and reviewed in every project review meeting.

Risk Mitigation Strategies
Avoidance
Eliminate the risk entirely — change scope, technology, or approach so the risk no longer applies.
Mitigation
Reduce probability and/or impact — early testing, prototyping, better training, phased rollout.
Transference
Shift impact to a third party — insurance, fixed-price contracts, SLAs with penalties.
Acceptance
Acknowledge the risk and set aside contingency reserves (time/cost buffer) to deal with it if it occurs.
Risk Register

Central artifact for proactive risk management. Maintained throughout the project lifecycle and reviewed regularly.

Risk ID
Unique identifier for tracking and reference.
Description
Clear statement of the risk event and its potential cause.
Probability
Likelihood rating (Low/Med/High or 0–1 numeric).
Impact
Consequence severity rating on cost, schedule, scope, or quality.
Owner
Team member responsible for monitoring and executing the mitigation plan.
Status
Open, Mitigated, Triggered, Closed.
